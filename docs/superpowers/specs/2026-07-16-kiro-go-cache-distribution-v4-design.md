# Kiro-Go 缓存分布 V4 设计

日期：2026-07-16

状态：已按用户审阅意见修订，待最终确认

## 1. 结论

本次只修改 Kiro-Go 的 Anthropic usage 分配模型，不修改 Sub2API 的缓存解析、定价、扣费、数据库或 Redis。

V4 模型的核心目标是：

- 正常可分配请求的 Token 加权缓存读取命中率以 `92.8%` 为中心，为超大受控读写请求给全量面板留出空间。
- 单条正常请求的读取命中率打散在 `85%–95%`。
- 正常请求中约 `70%–80%` 的请求创建缓存，中心值为 `75%`。
- 大请求提高缓存创建份额，利用 1 小时缓存创建与读取约 `1:20` 的 Token 费用关系压低绝对读取 Token。
- 小请求保持适中的创建份额，不为了降低少量读取而制造过大的创建记录。
- 成功但因 Kiro payload 限制发生截断的超大请求创建约 `100K–350K` 的 1 小时缓存，同时保留约 `20K–300K` 的受控缓存读取。
- 不设置读取或正常创建 Token 的硬上限，不产生大量相同平台值。
- 保留费用守恒、同步与流式一致性、重试幂等和单一 TTL 字段合同。

`92.8%` 是正常可分配请求窗口的中心目标，不要求每条请求都是 92.8%。Sub2API 全量面板会把正常请求和超大受控读写请求的 Token 一起计算，固定 500 条样本预计显示约 `85.0%`。

## 2. 当前模型

当前算法版本为：

```text
native-high-cache-v3
```

主要实现位于：

- `proxy/claude_usage_allocator.go`
- `proxy/cache_tracker.go`
- `proxy/handler.go`
- `proxy/translator.go`

当前正常请求使用以下读取命中率：

```text
target_hit_rate
= 0.952
 + 0.004 * size_factor
 + 0.001 * tool_ratio
 + 0.001 * stable_jitter

clamp(target_hit_rate, 0.95, 0.96)
```

当前约一半请求进入读写类别：

```text
hash % 100 < 50 -> 读取并创建
其余           -> 只读取
```

读写类别只在 4%–5% 的非读取部分中继续拆分创建：

```text
creation_fraction
= clamp(
    0.40
    + 0.20 * growth_ratio
    + 0.05 * tool_ratio
    + 0.05 * stable_jitter,
    0.40,
    0.65
  )
```

成功但 `payload.WasTruncated=true` 的请求不会进入分配器，最终保留普通输入 Token，缓存读取和创建均为 0。

## 3. 当前模型错在哪里

### 3.1 大请求的读取命中率方向相反

当前公式给 `size_factor` 正系数：

```text
+ 0.004 * size_factor
```

输入越大，目标命中率越接近 96%。由于缓存读取价格只有普通输入的 0.1 倍，费用守恒会把相同原始费用展开成大量读取 Token。因此，大请求同时具有更大的原始输入和更高的读取比例，绝对读取 Token 会被双重放大。

这正是当前记录中读取 Token 动辄几十万、七八十万甚至超过一百万的主要原因。

### 3.2 创建请求条数过少

当前设计类别为 50% 读写、50% 只读，但固定 500 条样本中，只有约 41.2% 的正常请求具有完整 TTL 创建明细。

创建请求条数偏少后，更多请求只能通过低价读取 Token 承担费用，进一步放大面板中的读取数量。

### 3.3 创建份额被压缩在过小的非读取区间

当前读取命中率为 95%–96%，非读取部分只有 4%–5%。即使创建占非读取部分的 40%–65%，创建占全部展示 Token 的比例通常也只有约 1.6%–3.25%。

这意味着当前公式表面上给了较高的 `creation_fraction`，实际创建 Token 仍然很小，无法有效替代读取 Token。

### 3.4 创建份额没有随请求规模增大

当前 `creation_fraction` 依赖增长比例、工具比例和稳定微扰，但不直接依赖输入规模。大请求没有获得更高的创建份额，无法利用费用关系主动压缩读取数量。

对于 1 小时缓存：

```text
读取价格 = 0.1x
创建价格 = 2x

1 个创建 Token 的费用
= 20 个读取 Token 的费用
```

在普通输入不变时：

```text
增加 1 个 1 小时创建 Token
可以减少约 20 个读取 Token
```

因此，为了压低大请求的绝对读取 Token，正确方向应当是提高大请求的创建份额，而不是降低创建份额。

### 3.5 超大截断请求被全部排除

`proxy/handler.go` 当前只有在以下条件成立时调用正常分配器：

```text
(finalUsageComplete || realInputTokens > 0)
&& !payload.WasTruncated
```

固定 500 条样本中有 68 条请求同时满足：

- 请求成功。
- 原始等价输入约 `101K–779K`。
- 缓存读取为 0。
- 缓存创建为 0。

这些记录与 `900 KiB` payload 截断保护路径高度吻合。它们不是因为 `High`、`XHigh` 或其他思考强度被排除，而是因为 payload 截断绕过了缓存分配器。

### 3.6 指标口径混在一起

当前生产版本 `0.1.156 / 8f2c4a91` 的 Sub2API 面板缓存命中率不是请求命中次数，也不是缓存读取与缓存创建的比值。该公式位于 `frontend/src/components/charts/TokenUsageTrend.vue`，实际计算为：

```text
sub2api_panel_hit_rate
= sum(cache_read_input_tokens)
  / sum(
      input_tokens
      + cache_creation_input_tokens
      + cache_read_input_tokens
    )
```

后端先在时间桶内汇总 Token，前端再计算比例。输出 Token 不进入分母。一条大请求会按 Token 数量显著影响结果，不存在“命中一次就加一分”的计数逻辑。

因此必须同时报告：

- 正常可分配请求读取命中率。
- 超大受控读写请求自身命中率、读取量和创建量。
- 与 Sub2API 代码完全一致的全量面板命中率。

## 4. 设计原则

V4 采用无外部状态的规模自适应模型：

- 不增加 Redis、数据库或新的缓存命名空间。
- 使用现有任务键、请求指纹和算法版本生成稳定类别和微扰。
- 同一请求重试得到相同类别和相同目标。
- 使用连续曲线和稳定微扰，不使用读取 Token 硬上限。
- 大请求同时降低读取命中率并提高创建份额。
- 超大截断请求进入独立的受控读写路径。

不采用进程内滚动窗口控制器。滚动控制器可以把正常短窗口锁得更接近 92.8%，但会引入并发状态、重启恢复和流量顺序依赖，不符合当前保持简单、便于维护的要求。

## 5. 请求分类

最终成功响应分为三类。

### 5.1 正常可分配请求

条件：

```text
最终 usage 有效
&& 原始输入和输出非负
&& payload.WasTruncated == false
```

进入正常 V4 读取和创建分配器，并在成功后提交 Prompt Cache Tracker 状态。

### 5.2 超大受控读写请求

条件：

```text
请求成功
&& 最终原始输入有效
&& payload.WasTruncated == true
```

行为：

- 生成约 `20K–300K` 的受控 `cache_read_input_tokens`
- 生成约 `100K–350K` 的 `ephemeral_1h_input_tokens`
- 不提交 Prompt Cache Tracker 状态
- 不把实际已截断的上游 payload 当成完整前缀保存

### 5.3 原始 usage 回退

以下情况不生成缓存读取或创建：

- 上游失败。
- 最终 usage 缺失。
- 原始 Token 为负数或溢出。
- 费用方程不存在合法整数解。
- 响应没有形成成功终态。

这些请求保留原始 usage，不提交状态。

## 6. 正常请求命中率模型

### 6.1 规模压力

继续使用当前对数压缩输入规模：

```text
size_factor
= clamp(log2(1 + raw_input_tokens) / 20, 0, 1)
```

再转换为规模压力：

```text
size_pressure
= smoothstep(0.60, 0.875, size_factor)
```

`size_pressure` 位于 0–1。小请求接近 0，当前正常样本中的最大请求接近 1。

### 6.2 命中率锚点

基础命中率使用以下锚点并执行平滑插值：

| `size_pressure` | 基础读取命中率 |
| ---: | ---: |
| 0.00 | 95.0% |
| 0.40 | 95.0% |
| 0.70 | 95.0% |
| 0.88 | 94.8% |
| 1.00 | 90.0% |

加入独立稳定微扰：

```text
target_hit_rate
= clamp(
    smooth_anchor_hit_rate
    + 0.010 * stable_hit_jitter,
    0.85,
    0.95
  )
```

其中 `stable_hit_jitter` 位于 `-1–1`。

该曲线的效果是：

- 小请求通常读取 94%–95%。
- 中请求仍保持较高读取，但不会全部集中在同一比例。
- 大请求逐步下降到约 89%–95%。
- 正常请求 Token 加权窗口中心约为 92.8%。
- 不对读取 Token 设置固定上限。

## 7. 创建请求条数

正常请求使用独立哈希生成创建概率：

```text
create_probability
= clamp(
    0.75 + 0.02 * stable_class_jitter,
    0.70,
    0.80
  )
```

再使用独立哈希抽样决定是否创建：

```text
stable_class_draw < create_probability
```

要求 `stable_class_jitter`、`stable_class_draw`、命中率微扰和创建份额微扰使用不同的哈希盐，避免变量之间产生无意相关性。

预期：

- 正常创建请求率约 75%。
- 合理观察窗口内位于 70%–80%。
- 所有正常请求都读取缓存。
- 未进入创建类别的请求为只读请求。

## 8. 创建 Token 份额

### 8.1 规模递增公式

创建类别使用：

```text
creation_fraction
= clamp(
    0.32
    + 0.38 * size_pressure^1.2
    + 0.035 * stable_creation_jitter,
    0.26,
    0.75
  )
```

`creation_fraction` 表示创建 Token 占非读取部分的比例，而不是占全部展示 Token 的比例。

效果：

- 小请求约 26%–36%，保持适中创建。
- 中请求随规模平滑增加。
- 大请求约 65%–75%，主动使用创建费用替代大量读取 Token。

V4 不再使用“预测创建超过 9K–20K 后压低创建份额”的反向软抑制。该抑制会在最需要创建费用的大请求上重新放大读取 Token，与本次目标相反。

### 8.2 目标份额

对于创建类别：

```text
read_share   = target_hit_rate
create_share = (1 - target_hit_rate) * creation_fraction
input_share  = 1 - read_share - create_share
```

对于只读类别：

```text
read_share   = target_hit_rate
create_share = 0
input_share  = 1 - read_share
```

## 9. 为什么提高创建能够降低读取

以 1 小时 TTL 为例，费用守恒为：

```text
20*T = 20*I + 2*R + 40*C1
```

设展示总量为：

```text
D = I + R + C1
```

在目标读取份额 `h` 和创建份额 `c` 固定时：

```text
D
= 20*T
  / (20*(1-h-c) + 2*h + 40*c)

R = h*D
```

在原始费用 `T` 和命中率 `h` 不变时，提高 `c` 会增大分母，因此展示总量 `D` 和绝对读取 Token `R` 都会下降。

这允许 V4 在保持正常窗口约 92.8% 读取命中率的同时，通过更高创建份额降低大请求的绝对读取 Token。

## 10. TTL 规则

正常请求继续复用现有稳定任务 TTL：

- 约 20% 任务使用 5 分钟创建。
- 约 80% 任务使用 1 小时创建。
- 同一任务和同一请求只能出现一种创建 TTL。

超大受控读写请求强制使用 1 小时创建：

```text
ephemeral_5m_input_tokens = 0
ephemeral_1h_input_tokens = oversize_creation_tokens
```

这样可以满足下游对 1 小时创建字段的计费需求，并利用 `1 创建 Token ≈ 20 读取 Token` 的费用关系压缩读取数量。

## 11. 超大受控读写模型

根据原始输入生成创建压力：

```text
oversize_pressure
= smoothstep(180K, 680K, raw_input_tokens)
```

目标读取量：

```text
read_target
= clamp(
    20K
    + 280K * oversize_pressure
    + 30K * stable_oversize_read_jitter,
    20K,
    300K
  )
```

目标创建量：

```text
creation_target
= clamp(
    100K
    + 250K * oversize_pressure
    + 25K * stable_oversize_creation_jitter,
    100K,
    350K
  )
```

读取和创建必须使用不同的哈希盐。读取目标为 `20K–300K`，创建目标为 `100K–350K`，中间值随原始输入平滑增加。

为防止普通输入变为负数，保留：

```text
input_reserve
= max(1K, 1% * raw_input_tokens)
```

1 小时费用守恒可写为：

```text
T = I + 0.1*R + 2*C1
```

先固定受控读取，再对创建目标执行费用可行性限制：

```text
C1
= min(
    creation_target,
    (T - 0.1*read_target - input_reserve) / 2
  )

R = read_target
I = T - 0.1*R - 2*C1
```

固定样本中最小超大请求的原始等价输入约为 `101.5K`。它无法同时严格满足创建至少 `100K`、读取至少 `20K`、普通输入至少 `1K`；费用守恒优先后，创建会降到约 `49.3K`。实现不得为了凑范围产生负数或破坏费用守恒。

这里的读取和创建范围是受控展示范围，普通输入保留量是费用合法性保护。超大请求仍不提交 Tracker 状态，因为实际发送给上游的 payload 已被截断。

## 12. 整数分配与费用守恒

继续使用当前无量纲权重：

```text
普通输入       = 20
缓存读取       = 2
5 分钟创建     = 25
1 小时创建     = 40
```

正常请求必须严格满足：

```text
20*T
= 20*I
  + 2*R
  + 25*C5
  + 40*C1
```

并满足：

```text
C5 == 0 或 C1 == 0
cache_creation_input_tokens = C5 + C1
```

现有候选搜索器继续负责整数化，但需调整：

- 读取份额合法范围从 95%–96% 改为 85%–95%。
- 普通输入份额上限从 5% 扩大到 15%，支持大请求低命中率和只读类别。
- 创建份额允许达到非读取部分的 75%。
- 候选排序优先命中率误差，其次创建份额误差，再次普通输入误差。
- 创建类别在极小输入下无法生成合法正整数创建时，可降级为只读类别。
- 任意负数、溢出、TTL 混用或费用不守恒候选必须拒绝。

## 13. 状态与幂等

稳定哈希种子继续包含：

```text
task_key
+ request_fingerprint
+ prompt_cache_algorithm_version
+ variable_salt
```

算法版本升级为：

```text
native-high-cache-v4
```

正常成功请求：

- 同一请求指纹复用已提交 usage。
- 新请求成功后提交前缀和 usage。
- 重试不重复推进成功轮次。

超大截断请求：

- 使用请求指纹分别生成稳定读取量和创建量。
- 不提交前缀或 usage 到 Tracker。
- 避免把实际已截断 payload 当成完整缓存前缀。

## 14. 响应字段合同

同步响应和流式最终 `message_delta.usage` 都必须返回完整标准字段：

```json
{
  "input_tokens": 0,
  "output_tokens": 0,
  "cache_read_input_tokens": 0,
  "cache_creation_input_tokens": 0,
  "cache_creation": {
    "ephemeral_5m_input_tokens": 0,
    "ephemeral_1h_input_tokens": 0
  }
}
```

要求：

- `cache_creation_input_tokens` 始终等于两个 TTL 明细之和。
- 5 分钟和 1 小时字段不能同时大于 0。
- 超大受控读写请求必须同时返回缓存读取和 1 小时创建字段。
- Sub2API 继续只负责标准字段解析、定价和扣费。

## 16. 指标与验收口径

### 16.1 主指标

只统计正常可分配请求：

```text
normal_window_hit_rate
= sum(cache_read_input_tokens)
  / sum(
      input_tokens
      + cache_read_input_tokens
      + cache_creation_input_tokens
    )
```

目标中心：

```text
92.8%
```

固定 500 条回放验收范围：

```text
92.3%–93.2%
```

生产真实流量因请求规模组合变化，500 条滚动观察窗口允许：

```text
92%–93.5%
```

V4 不通过引入滚动状态强行锁定 92.800%。

### 16.2 Sub2API 全量面板指标

严格复用 Sub2API 当前前端公式：

```text
sub2api_panel_hit_rate
= sum(cache_read_input_tokens)
  / sum(
      input_tokens
      + cache_creation_input_tokens
      + cache_read_input_tokens
    )
```

该指标包含正常请求和超大受控读写请求。固定 500 条回放预计约为：

```text
85.0%
```

固定回放验收范围为 `84.5%–85.5%`。生产真实流量中必须作为正式观察指标报告；不继续追求 90%，否则超大请求读取会重新上升到数百万 Token。

### 16.3 条数指标

正常请求单独计算：

```text
normal_read_request_rate
normal_create_request_rate
```

要求：

- `normal_read_request_rate` 接近 100%。
- `normal_create_request_rate` 位于 70%–80%。

全部请求另外计算：

```text
overall_create_request_rate
oversize_controlled_read_write_rate
```

同时报告超大请求的读取、创建和自身命中率分位数，避免全量面板比例掩盖绝对 Token 异常。

## 17. 测试设计

### 17.1 命中率单元测试

- 所有正常目标命中率位于 85%–95%。
- 小请求基础命中率高于大请求。
- 锚点间连续且单调，不产生跳变。
- 同一请求指纹得到相同命中率。

### 17.2 创建分布测试

- 创建请求率在大样本中位于 70%–80%。
- 大请求的 `creation_fraction` 高于小请求。
- 小请求创建份额位于适中范围。
- 大请求创建份额可达到约 65%–75%。
- 不存在正常创建 Token 固定平台值。

### 17.3 费用守恒测试

- 5 分钟和 1 小时创建分别严格守恒。
- 同一请求不能同时产生两种 TTL。
- 创建增加后绝对读取 Token 按预期下降。
- 整数取整后仍完全满足费用方程。

### 17.4 超大请求测试

- 成功截断请求生成 20K–300K 的受控读取。
- 成功截断请求生成约 100K–350K 的 1 小时创建，并受费用可行性限制。
- 读取量和创建量随原始输入平滑增加，并使用独立哈希盐。
- 普通输入保留量始终满足约束。
- 极小费用预算发生冲突时优先保证费用守恒和非负输入。
- 超大请求不提交 Tracker 状态。
- 失败或 usage 缺失请求不生成缓存字段。

### 17.5 合同测试

- 同步和流式最终 usage 完全一致。
- 创建汇总等于 TTL 明细之和。
- 所有标准字段始终存在。
- 重试复用相同 usage。

### 17.6 分布回放测试

- 使用固定 500 条快照重放输入规模。
- 使用至少 10,000 个真实格式的稳定任务键和请求指纹验证哈希分布。
- 正常加权命中率位于 92.3%–93.2%。
- Sub2API 全量面板命中率位于 84.5%–85.5%。
- 正常创建请求率位于 70%–80%。
- 读取 P95 和最大值明显低于 V3 基线。
- 大请求创建平均显著高于小请求。

## 18. 部署与观察

本设计不包含独立 UAT 流量门槛。实施完成后先执行本地单元测试、合同测试和固定样本回放；部署属于后续独立操作，必须按生产变更流程获得明确授权。

部署后使用新产生的真实记录观察：

- 正常 500 条加权命中率。
- 正常创建请求率。
- 全部创建请求率。
- Sub2API 全量面板命中率。
- 超大受控读写比例。
- 超大读取和创建 Token 分位数。
- 读取 Token P50、P95 和最大值。
- 小、中、大请求的创建 Token 分位数。
- 1 小时创建字段完整率。

不回写或重算历史 usage。

## 19. 回滚边界

- V4 只修改 Kiro-Go。
- 回滚只需要恢复上一版 Kiro-Go 镜像。
- 不修改 Sub2API 数据库和 Redis。
- 不重建 PostgreSQL、Redis、Caddy、CPA 或其他无关服务。
- 新旧算法产生的历史 usage 均保持原样。

## 20. 最终方案摘要

V3 的根本问题不是单纯“命中率太高”，而是三个因素叠加：

1. 大请求反而获得更高读取命中率。
2. 创建请求条数和创建份额不足。
3. 超大截断请求完全没有创建缓存。

V4 的修正方向是：

1. 正常窗口命中率从约 95.6% 调整到约 92.8%。
2. 单条请求按规模打散到 85%–95%。
3. 创建请求率提升到约 75%。
4. 大请求显著提高创建份额，小请求保持适中。
5. 超大截断请求改为 20K–300K 受控读取和约 100K–350K 的 1 小时创建。
6. 继续保持费用守恒和标准 Anthropic usage 字段合同。

该模型直接针对绝对读取 Token 过大的问题，同时保持实现无外部状态、可测试、可回滚和仅修改 Kiro-Go 的边界。
