# Kiro-Go 缓存读取覆盖率与读写倍数微调设计

日期：2026-07-14

状态：已确认

## 1. UAT 证据

二号机 `154.36.172.65` 当前运行提交
`1b94c33d64aca3d1c7412c3f057d1fe934a774cd`。

截至 2026-07-14 13:44，切换后 account `1910` 共记录 422 条成功 usage：

```text
只创建：243
读取加创建：177
只读取：0
无缓存：2
读取请求占比：41.94%
创建请求占比：99.53%
```

连续 10 条记录中的读取条数最少为 `0`、平均为 `4.237`、最多为 `9`。

全部记录累计读取/创建 token 倍数为 `2.9394`，但有读取记录的单条倍数为：

```text
最小：3.9014
中位数：4.1752
平均：6.2329
最大：11.2491
```

因此问题有两部分：

1. 进入读取分支的请求条数太少。
2. 少数读取请求的读取/创建倍数过高。

## 2. 根因

当前首轮和过期重建阶段固定返回：

```text
read_share = 0
create_share = cache_total
```

当前续轮固定限制：

```text
read_share = 78% 到 90%
create_share = 8% 到 20%
```

第二组边界天然产生约 `3.9` 到 `11.25` 倍的读取/创建比例。

## 3. 调整目标

保留现有请求分析、任务键、前缀匹配、状态表、TTL 和整数费用守恒实现，只调整目标比例函数。

- 所有现有续轮继续进入读取加创建分支。
- 首轮和过期重建按稳定请求特征分流：
  - 约 `80%` 进入读取加创建分支。
  - 约 `20%` 保持只创建。
- 预计按当前 UAT 的首轮/续轮构成，整体读取请求占比由约 `42%` 提升到约
  `88%`。
- 所有读取加创建目标严格满足：

```text
2.0 <= cache_read / cache_creation <= 5.0
```

- 代表性分布平均读取/创建倍数位于 `2.8` 到 `3.2`。
- 原始输入越大，在其他特征相同时读取倍数越高。
- 稳定微扰继续打散比例。

## 4. 最小实现

首轮读取门禁使用已有 `stable_jitter`：

```text
phase == continue
或
stable_jitter >= -0.60
```

`stable_jitter` 近似均匀分布于 `-1` 到 `1`，因此约 `80%` 的首轮和重建请求进入读取分支。

读取加创建的目标倍数：

```text
ratio_raw
= 2.40
+ 0.95 * size_factor
+ 0.25 * reuse_ratio
- 0.15 * growth_ratio
+ 0.35 * stable_jitter

ratio = clamp(ratio_raw, 2.0, 5.0)
```

目标比例由倍数直接反解：

```text
cache_total = 1 - input_share
create_share = cache_total / (1 + ratio)
read_share = cache_total - create_share
```

最终整数求解后再次校验实际 token 倍数位于 `2` 到 `5`。

## 5. 保留约束

- 缓存总比例继续位于 `95%` 到 `99%`。
- 每条请求只使用 `5m` 或 `1h` 一种创建 TTL。
- 长期 TTL 分布继续约为 `20% / 80%`。
- `cache_creation_input_tokens` 等于两个 TTL 明细之和。
- 输入侧费用继续整数完全守恒。
- 输出 token 不变。
- 同步和流式最终 usage 使用同一分配函数。
- 相同请求指纹的幂等重试继续复用相同 usage。
- 候选搜索继续不超过 `64`。
- 无合法解时继续回退原始 usage。

算法版本提升为：

```text
native-high-cache-v2
```

## 6. 本轮不做

- 不修改任务键。
- 不增加公共前缀命名空间。
- 不增加缓存状态或服务器锁。
- 不增加只读取模式。
- 不删除现有缓存跟踪器。
- 不修改 Sub2API。

只读取模式在维持费用守恒和 `95%` 到 `99%` 缓存率时会显著放大展示 token，
不适合作为本轮快速 UAT 的参数微调。
